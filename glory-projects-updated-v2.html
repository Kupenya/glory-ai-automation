Invoice Inbox Automation
— Novus Realty

A fully automated invoice processing pipeline that watches a Gmail inbox, extracts structured financial data from any invoice — whether it's a PDF attachment or plain email text — using GPT-4o vision, and logs everything into a Google Sheet. Zero manual data entry.

Make GPT-4o Vision Gmail API PDF.co Google Sheets JSON Parser
2
processing paths (PDF + body)
0
manual data entry steps
1
idempotency gate — no duplicates
Overview
The problem was not invoice volume. It was invoice variability.
Novus Realty's finance intake lived inside a Gmail inbox. Some invoices arrived as PDFs. Some were pasted into the email body. Some were multi-page. Some were forwarded twice. The team was manually reading each one, pulling out dates, vendors, invoice numbers, totals, and then copying everything into a spreadsheet. The work was repetitive, slow, and easy to get wrong.
The real issue was not just data entry. It was the lack of a reliable intake system. There was no consistent way to process different formats, no protection against duplicate logging, and no clean audit trail showing which invoice had already been handled.
The challenge
C1
Input Variability
The same business document arrived in completely different shapes
Some invoices were clean PDFs. Others were screenshots inside a PDF. Others were plain text in the email body. A useful system could not depend on a single format or a single extraction method.
C2
Data Integrity
A bad automation would be worse than manual work
If the workflow duplicated invoices, wrote malformed rows, or silently missed multi-page documents, the spreadsheet would stop being trustworthy. Accuracy mattered more than speed.
The solution
S1
Dual-path extraction
One intake system, two extraction routes
The workflow first decides what kind of invoice it received. PDFs are converted to images and passed through GPT-4o Vision. Email-only invoices are parsed directly from the message body. Both routes converge into the same JSON schema, so downstream processing stays consistent even when the inputs do not.
S2
Trust boundary
Nothing reaches the sheet unless it survives validation
The model is used for extraction, not for authority. JSON-only output, parser validation, a required-data gate, and the Gmail processed label together turn a loose LLM workflow into a controlled intake pipeline.
Full workflow flow
Gmail Watch → List Attachments → Gate: Not Processed? → Router → PDF path: PDF.co → Images → GPT-4o Vision | Body path: GPT-4o → JSON Parse → Gate: Has Data? → Google Sheets Row → Label Email
01
Trigger
Gmail inbox watch — keyword filtered
The trigger watches the Gmail inbox every cycle for emails containing "invoice", "billing", or "statement" in the subject or body. This isn't a catch-all — filtering at the trigger level means the workflow only runs on relevant mail, not every email that lands in the inbox. Marks emails as seen on watch to avoid re-processing reads.
02
Idempotency Gate
Label check — "Not Already Processed"
Before doing anything expensive, the router checks whether the email already has the "Invoice Processed" label applied. If it does, the entire route is skipped. This is the idempotency gate — it prevents the same invoice being logged twice if the scenario is re-triggered or re-run.
✓
Pass: No "Invoice Processed" label → continue to extraction
✕
Fail: Already labelled → route exits immediately, no duplicate
03
AI Extraction — PDF Path
PDF → Images → GPT-4o Vision extraction
When the email has a PDF attachment: PDF.co converts the PDF into image files (JPG, inline). The iterator then feeds each page image one at a time into GPT-4o with a structured system prompt instructing it to act as an invoice extraction assistant with vision capabilities. The model returns JSON only — vendor name, invoice number, invoice date, due date, total amount, currency, and line items. The JSON parser then validates the output before anything touches the spreadsheet.
04
AI Extraction — Body Path
Email body → GPT-4o text extraction
When there's no PDF attachment, the second route reads the raw email body and sends it to GPT-4o with the same structured extraction prompt — JSON only, no explanation, no markdown. Same fields extracted, same JSON parser validates the output. Both paths produce identical structured data downstream.
05
Output Gate
Data quality gate before logging
Before writing to Google Sheets, a filter checks "Has Invoice Data" — meaning the JSON parse succeeded and returned the expected fields. If GPT-4o returned something malformed or empty, this gate stops the write. This prevents blank or broken rows from polluting the sheet.
✓
Pass: Valid JSON with required fields → add row to Sheets
✕
Fail: Malformed output → row skipped, no bad data written
06
Output + Label
Google Sheets row + "Invoice Processed" label
The extracted data is appended as a new row in Google Sheets — received date, vendor name, invoice number, invoice date, due date, total amount, and currency all mapped to their columns. After the successful write, the Gmail label "Invoice Processed" is applied to the email. This closes the idempotency loop: any future trigger of this email will be stopped at Step 02.
Edge cases I designed for
🔄
Duplicate processing
Built the Gmail label gate first before anything else. Idempotency isn't optional when workflows re-trigger on schedule.
📐
AI output reliability
Forced JSON-only responses and added a parse + validation gate before any write. LLMs hallucinate — I don't let that reach the spreadsheet.
📄
Multi-page PDFs
Used PDF.co's iterator to process each page image separately, so multi-page invoices aren't truncated or missed entirely.
Outcomes
The outcome is a finance intake process that behaves like a system, not an inbox habit. Every qualifying invoice is captured, parsed, validated, written once, and marked as processed. The spreadsheet stays clean, the audit trail stays visible in Gmail, and the team no longer spends working hours retyping information that already exists.

Project 02 — Make · Airtable · Google Sheets

Reporting & Dashboard
Automation

A webhook-triggered multi-department reporting engine that pulls raw data from Google Sheets and Airtable, computes KPIs for Sales, Project Delivery, and People Ops, then upserts clean monthly summaries back into an Airtable dashboard — with error state logging when source data is absent.

Make Google Sheets Airtable Webhook Aggregators Set Variables
3
departments processed in parallel
Upsert
create-or-update, never duplicates
Auto
error state logged when no data
Overview
Three departments were reporting from three different realities.
Sales performance lived in Google Sheets. Delivery health lived in Airtable. People Ops had its own records and timing. Every reporting cycle meant someone had to gather the numbers manually, reconcile them by hand, and hope the final dashboard still matched reality by the time leadership looked at it.
The real problem was not dashboard design. It was the lack of a reporting system that could compute trustworthy monthly KPIs across functions, write them back into one place, and show an honest state even when a source returned nothing.
The challenge
C1
Execution model
The reporting job was slow, but the trigger still needed to feel reliable
A single run touches multiple data sources, performs grouping, computes KPIs, and then writes results back into Airtable. Holding the incoming webhook open for all of that would create fragile integrations and timeout risk.
C2
Truthfulness
A stale dashboard is worse than an empty one
If a source returned zero rows, the system could not just leave the old numbers in place and pretend the run succeeded. Missing data had to be surfaced explicitly so the dashboard reflected the state of the process, not just the last successful run.
The solution
S1
Async reporting engine
Acknowledge immediately, then compute in the background
The webhook returns a 200 response first, then the workflow fans out into three reporting routes for Sales, Project Delivery, and People Ops. That keeps the caller fast and stable while letting the heavy reporting work finish safely behind the scenes.
S2
Upsert + status logging
Each month and department resolves to one clean dashboard record
Each route aggregates by unique month, computes KPIs, then searches for the matching Airtable record before writing. If it exists, update it. If it does not, create it. If no data exists, write a clear No Data status with a timestamp instead of leaving a stale number behind.
Full workflow flow
Webhook Trigger → Instant 200 Accept → 3-Route Router → Sales: Sheets → Aggregate → KPIs | Project Delivery: Airtable → Late calc → KPIs | People Ops: Airtable → Hire metrics → Gate: Has Data? → Compute KPIs per unique month → Airtable Upsert
01
Trigger + Async Pattern
Webhook trigger with immediate 200 response
The scenario is triggered via a custom webhook. The very first action is to respond 200 "accepted" to the caller — before any processing begins. This is intentional: reporting runs are slow (multiple Airtable reads, aggregations), and holding the HTTP connection open would cause timeouts. The caller gets immediate confirmation; the work runs asynchronously behind it.
02
Route A — Sales
Sales KPI computation from Google Sheets
Filters Google Sheets rows for 2025 data, aggregates into memory, then uses distinct(map()) to get unique reporting months. For each month: total leads, closed won count, conversion rate, revenue, and average deal size are computed. A router then checks if Airtable already has a record for that month/department — update if found, create if not.
03
Route B — Project Delivery
Project health metrics with late delivery detection
Reads project records from a separate Airtable base. For each project, computes on-time vs late delivery by comparing Actual_End_Date to Planned_End_Date. Also tracks scope creep rate (Is_Scope_Creep field), budget variance (actual vs budgeted cost), and total project count. Same unique-month loop and upsert pattern as Sales.
04
Route C — People Ops
HR metrics — headcount, attrition, time-to-hire
Reads the People Ops Airtable base. Computes days to hire by subtracting Application_Date from Offer_Accept_Date (in milliseconds, then divided to days). Tracks new hires, exits, roles filled, and total headcount per month. Notably, this route uses inverted router logic — "No Data" is Route A, good data is Route B, so the empty state is caught first.
05
Error State Handling
Empty source → log "No Data" status to Airtable
Each of the three routes has a "No Data" fallback. If the source returns 0 records for a given period, instead of silently failing, the workflow writes an error state back to the Airtable dashboard record: status = "No Data", error_message = "Source returned 0 records at [timestamp]". This means the dashboard always reflects the state of the last run — success or failure — rather than just going stale with old numbers.
✓
Has data: Compute KPIs → upsert record (create or update by department + month)
✕
No data: Write "No Data" status + timestamp to specific Airtable record — dashboard still shows something honest
Edge cases I designed for
⏱️
HTTP timeout risk
Sent the 200 response before any processing. Reporting runs are slow — holding the connection open would fail the caller.
🔁
Duplicate records
Used upsert logic everywhere: search first, update if found, create if not. Running the scenario twice never creates two records for the same month.
📉
Silent empty runs
Built explicit "No Data" logging so the dashboard always reflects the last run's outcome — the worst thing a reporting system can do is silently lie with stale numbers.
Outcomes
The reporting cycle moves from manual assembly to repeatable computation. Leadership gets one Airtable dashboard that can be refreshed on demand, each department's numbers are written back without duplication, and missing-data runs become visible states instead of silent dashboard drift.

Project 03 — n8n · OpenAI · Google Docs/Drive · Airtable · Slack

AI-Powered Proposal
Automation System

Two-phase end-to-end system. Phase 1 reads prospect data from a Google Sheet, verifies emails, generates a personalised AI sales proposal via GPT-4o-mini, creates a branded Google Doc, and alerts the team on Slack. Phase 2 listens for the approval decision and dispatches the final PDF directly to the client via Gmail.

n8n GPT-4o-mini Hunter.io Google Docs Google Drive Airtable Slack Gmail
2
linked workflow phases
5+
conditional gates in Phase 1
0
manual proposal writing steps
Overview
Proposal writing was consuming senior sales time that should have been spent selling.
The manual process was always the same: check the lead details, validate the contact, research the company, write tailored proposal copy, format the document, then send it out. It was high-effort work with too many handoffs and too many opportunities for inconsistency.
The business needed a system that could generate strong first drafts automatically, keep the proposal branded and structured, and still preserve a human approval point before anything reached a prospect.
The challenge
C1
Personalization
Automating proposals is easy. Automating proposals that still feel bespoke is not.
A generic draft saves time but weakens trust. The system had to preserve company-specific language and a credible value proposition without forcing someone to rewrite everything from scratch.
C2
Control
The last mile could not be fully autonomous
Proposals are high-stakes sales assets. The workflow needed strong gates for invalid emails, duplicate runs, and repeat sends, plus a clear human approval checkpoint before delivery.
The solution
S1
Two-phase system
Generation and delivery are split on purpose
Phase 1 handles ingestion, deduplication, email verification, AI proposal generation, document creation, and internal alerting. Phase 2 does nothing until a human approves the draft, then downloads the final PDF and sends it to the client.
S2
Operational safeguards
Every expensive or risky step is earned
No proposal is generated for invalid emails. No duplicate file is created if one already exists. No client email is sent without an explicit Slack approval event. The result is faster sales ops without surrendering control.
Phase 1 — Generation
Schedule Trigger → Google Sheets Read → Normalize Company Name → Remove Duplicates → Hunter.io Verify Email → Valid Email? → Airtable Upsert Lead → AI Draft Exists? → GPT-4o-mini Generate Proposal → File ID Exists? → Copy Template Doc → Replace Placeholders → Log to Airtable → Slack Alerts
1.1
Data Prep
Normalize + deduplicate before anything expensive
The first thing the workflow does after reading the Google Sheet is normalize the company name — lowercase, strip whitespace — then run Remove Duplicates on [Client Email, cleanCompanyName]. This means even if someone submits the same prospect twice with slightly different formatting, only one record flows through. The email verification API (Hunter.io) costs money per call — deduplicating before that step avoids wasted spend.
1.2
Email Verification Gate
Hunter.io email verifier — hard gate
Each email address is verified against Hunter.io's deliverability API. If the status comes back invalid, the record is skipped entirely — no proposal generated, no file created, no Slack notification.
✓
Valid: Upsert to Airtable, continue to proposal generation
✕
Invalid: Route exits — no proposal generated for undeliverable addresses
1.3
AI Generation
GPT-4o-mini: senior B2B sales strategist prompt
The AI generation step uses a detailed system prompt positioning GPT-4o-mini as a senior B2B sales strategist and copywriter. It receives the client name, company name, and company context and returns a structured proposal text with personalised approach, value proposition, and call to action — all in the client's voice. This step is gated: if an AI draft already exists in Airtable for this lead, the generation is skipped entirely and the existing draft is used. Prevents re-generating proposals for the same lead on subsequent runs.
1.4
Document Creation
Copy template → Replace placeholders → Log link
Google Drive copies a master proposal template document into a client-specific file. The Google Docs update step then does a replaceAll operation on placeholder tokens like {{AI_Approach}} — replacing them with the actual AI-generated content. Again gated: if the Google Doc already exists (Internal File ID in Airtable), the copy step is skipped and the existing doc is updated in place. The final proposal link is logged back to Airtable.
1.5
Notifications
Dual Slack channel alerts — critical + decisions
Two Slack messages fire at the end of Phase 1: one to #proposals-alerts-critical (operational visibility — the proposal is ready) and one to #proposals-decisions (the message a human uses to review and approve the proposal for sending). The decisions channel message uses Block Kit formatting with action buttons — a "Send" button triggers the Phase 2 webhook. This is the human checkpoint in the loop: no proposal goes to a client without a person seeing it first.
Phase 2 — Delivery
Webhook (Slack Button) → Get Airtable Record → Download Google Doc as PDF → Gmail — Send PDF to Client → Update Airtable (sent status) → Queued check → Slack #proposal-sent
2.1
Webhook Trigger
Human approval fires the webhook
Phase 2 starts when the Slack Block Kit button in #proposals-decisions is clicked. This fires a webhook carrying the lead's record ID. The workflow fetches the full lead record from Airtable — including the Proposal Link (the Google Doc URL) and all client contact details.
2.2
File + Delivery
Download as PDF → send via Gmail
Google Drive downloads the proposal Doc as a PDF (using the file ID extracted from the proposal link URL). Gmail sends it to the client with a personalised subject line ("Proposal for [Company Name]") and greeting using the client's actual name. The email is professional and human-readable — not a bot-looking notification.
2.3
State Update + Notification
Airtable update + dual Slack confirmation
After dispatch, Airtable is updated with sent status and timestamp. Slack receives confirmations in both #proposals-decisions (the approver sees it was sent) and #proposal-sent (team-wide visibility). The entire delivery trail — who approved, when it was sent, which file was attached — is logged in Airtable for audit.
Edge cases I designed for
🛡️
No accidental re-sends
Every expensive step (AI generation, file creation) is gated on whether it already happened. Re-running Phase 1 never overwrites a proposal or sends a duplicate to a client.
👤
Human in the loop
Phase 2 only fires when a human clicks approve in Slack. AI drafts the proposal — a human decides whether it goes out. That's the right boundary.
💸
API cost control
Normalized and deduplicated before calling Hunter.io. Gated AI generation before calling OpenAI. Every external API call is earned, not automatic.
Outcomes
The sales workflow gets its leverage back. Draft creation, document assembly, and delivery prep happen automatically, but the final send still belongs to a human approver. That means faster turnaround, more consistent proposal quality, and a complete audit trail without giving up control at the moment that matters.

Project 04 — n8n · Airtable · Apify · OpenAI · Bouncer

Lead Generation &
Outreach Automation

A fully stateful, persona-driven lead machine. Define an ICP in Airtable, trigger the workflow, and it discovers prospects via Apify, enriches them with LinkedIn profile data and company website scraping, verifies emails via Bouncer, and generates personalised outreach copy (email + LinkedIn message) using GPT-4o-mini. Every step tracks its own state.

n8n GPT-4o-mini Apify (2 actors) Bouncer Airtable HTTP Request Code nodes
6
status stages tracked in Airtable
2
Apify actors (discovery + enrichment)
Resume
run logic — never loses progress
Overview
Manual lead generation does not break because the team lacks effort. It breaks because the work does not compound.
Every new campaign used to start from zero: define the persona, hunt for leads, research companies, validate addresses, write copy, and repeat. It was slow, inconsistent, and hard to scale because too much of the process depended on human memory and manual research.
This system turns prospecting into a repeatable pipeline. A user defines the target persona once. From there, the workflow discovers leads, enriches them, validates the contact path, and produces grounded outreach that is ready for review. The high-level design aligns with the uploaded one-pager. fileciteturn2file12
The challenge
C1
Pipeline state
Long-running lead generation needs memory, not just automation
Prospecting runs can fail midway, return partial data, or be re-triggered with the same persona. Without resumability and deduplication, the system would either restart unnecessarily or create a messy Airtable base full of duplicates.
C2
Grounding
Personalized outreach is only useful if it is grounded in real company context
Scraped leads are rarely complete. Some companies have usable About pages. Some do not. Some prospects have valid emails. Some only have LinkedIn. The workflow had to adapt instead of failing on imperfect data.
The solution
S1
Structured lead states
Airtable becomes the operating system for the run
Each persona run moves through explicit states, existing runs can be resumed, and every lead is classified based on what was actually found: complete, review-needed, or outreach-ready. The system can be rerun safely without corrupting the data set. fileciteturn2file12
S2
Fallback-first enrichment
Discover, enrich, validate, then generate
Apify handles discovery, LinkedIn and website crawls provide context, Bouncer validates deliverability, and GPT only writes outreach when there is enough real information to justify personalization. When email is missing, the workflow downgrades gracefully to LinkedIn-first outreach instead of collapsing. fileciteturn2file12
Full workflow flow
Airtable Poll (1 min) → Status = Queued? → Prepare Persona Context → Existing Run? → Status → Searching → Apify Leads Finder → Normalize Leads → Min Data? → Lead Exists? → Status → Enriching → Apify LinkedIn Scraper → Apify Website Crawler → Status → Validating → Email? → Bouncer Verify → Qualified? → GPT-4o-mini Outreach → Airtable Outreach Record → Run Summary
01
Trigger + Status Gate
Airtable polls every minute — only picks up "Queued"
The workflow polls the Personas table in Airtable every minute. The first conditional gate checks status = "Queued" — only records explicitly queued for processing flow through. Records in any other status (Searching, Enriching, Done) are ignored, preventing the workflow from re-processing a run already in progress.
02
Run Resumability
Resume logic — never starts from scratch
Before creating a new run record, the workflow searches Airtable for an existing run linked to this persona. If one exists, it resumes from that run's context. If not, it creates a new run record. This means if the workflow is interrupted mid-run, it picks up where it left off — it doesn't re-search for leads or re-enrich contacts already processed.
03
Discovery
Apify leads-finder — ICP-targeted search
The persona context (job title, industry, location, company size) is passed to Apify's leads-finder actor which searches for matching prospects across LinkedIn and other sources. Raw results are normalized in a code node: URL sanitization, null handling, field mapping. Each lead is checked for minimum data requirements (full_name + linkedin_url) before being processed further — incomplete leads are flagged separately, not silently dropped.
04
Lead Deduplication
Check existing leads before creating new records
Before creating an Airtable lead record, the workflow searches for an existing lead with the same LinkedIn URL. If found, it attaches the existing record ID and updates it. If not, it creates a new record. This prevents the leads table from filling up with duplicates when the same prospect appears across multiple runs. Update wins over insert.
05
Enrichment — Two Apify Actors
LinkedIn profile + website crawler → company context
Two enrichment steps run in sequence. First: Apify's LinkedIn profile scraper pulls the prospect's headline, current role, summary, and company details. Second: Apify's website crawler hits the company's About page and homepage. A code node then builds a structured company context string from both sources — this context is what the AI outreach generator uses. Rich context = better copy.
06
Email Validation Gate
Bouncer email verifier — with LinkedIn fallback
If the lead has an email address, it's sent to Bouncer's verification API. The result maps to a validation status: deliverable, risky, or undeliverable. Undeliverable addresses get lead_status = "Needs Review". If no email is present at all, a skip node sets validation_status = "Skipped" and flags the lead as LinkedIn-first outreach only — the workflow doesn't stop, it adapts.
✓
Deliverable: Proceed to copy generation
?
No email: Skip validation → LinkedIn-first copy only
✕
Undeliverable: Lead marked "Needs Review" — no copy generated
07
AI Outreach Generation
GPT-4o-mini — personalised email + LinkedIn copy
Only leads that pass the qualification gate (website_status = Valid) reach this step. The AI receives the full enriched company context, the prospect's LinkedIn data, and the persona's target messaging to generate two pieces of outreach copy: a personalised cold email and a LinkedIn connection message. Both are parsed from JSON and stored in the Outreach table in Airtable — ready for review or direct use.
08
Run Summary
End-of-run metrics logged to Airtable
A code node at the end of the run aggregates counts from all previous steps: total leads discovered, leads ready for copy, leads needing review, incomplete leads. These are written to the run record in Airtable so you can see, at a glance, how each run performed without having to count records manually. The persona status also flips to "Done."
Edge cases I designed for
💾
State persistence
Every status change is written to Airtable immediately. If the workflow crashes mid-run, the state is not lost — it can resume or be inspected.
🧹
Data quality before AI
Minimum data gates and website validation ensure the AI only generates copy for leads that are actually worth reaching out to. No copy for bad data.
📬
No wasted sends
Email verification before copy generation means we never craft an outreach message for an address that will bounce. Bouncer catches this before it becomes a deliverability issue.
Outcomes
The result is a prospecting pipeline that produces usable output, not just raw records. Discovery, enrichment, validation, and message generation happen in sequence, and every run leaves behind enough state to review quality, resume safely, and compare ICP performance over time. That is what makes it scalable.

Project 05 — n8n · OpenAI · LinkedIn · Twitter/X · Gmail · Jina.ai

Fetemi Content Ops
— Full-Stack Content System

A two-workflow content production and publishing system with a REST API layer. Submit an idea or URL → get three AI-drafted articles → pick one → get platform-adapted versions for LinkedIn, Twitter, and email newsletter → publish or schedule across all channels in one call. Fully stateful, idempotent, and resumable at any stage.

n8n GPT-4o GPT-4o-mini Jina.ai LinkedIn API Twitter/X API Gmail n8n Data Table Webhook API
5
API endpoints across 2 workflows
3
platforms published simultaneously
Hash
based deduplication — never re-generates
Overview
Content operations were bottlenecked by repetitive format work, not by lack of ideas.
One strong idea still had to be researched, shaped into an article, rewritten for each platform, scheduled, and then tracked after publication. The slow part was not creativity. It was the operational overhead that came after the idea already existed.
Fetemi needed a system that could accept either a raw idea or a URL, generate multiple viable long-form drafts, let a human choose the strongest angle, adapt that choice into platform-specific content, and publish safely without double-posting.
The challenge
C1
Input diversity
The same pipeline had to work for both raw ideas and external URLs
URL inputs need fetching, cleanup, and summarization before generation. Idea inputs do not. The workflow had to normalize both into one durable generation state without branching into chaos.
C2
Safe publishing
Repurposing is only valuable if retries do not create duplicate posts
Publishing across channels introduces partial-failure risk. If one channel succeeds and another fails, the system needs enough state to retry only what is missing.
The solution
S1
Idempotent content engine
Hash the input, cache the run, keep the state durable
Every submission is normalized and hashed before generation. If the same input appears again, the system returns the existing run instead of re-spending tokens. All content state lives in the Data Table, so execution history, selected draft, adapted assets, and publish status are queryable at any time.
S2
Human choice, automated distribution
Generate options first, adapt only the chosen angle
The workflow produces three distinct article drafts, waits for a deliberate selection, then adapts the winning version into LinkedIn, Twitter/X, and email formats. Per-channel publish gates make retries safe and keep one failed post from forcing a full rerun.
Workflow 1 — Content Automation (3 endpoints)
Endpoint 1: /content-input-v4 — Draft generation
POST /content-input-v4 → Validate type + content → Build executionKey + inputHash → Hash exists? → return cached → URL or Idea? → URL: Jina.ai fetch → GPT-4o-mini summarise → SEO keywords (GPT-4o-mini) → 3 drafts (GPT-4o) → Insert content_runs row → Respond: drafts_ready
5.1
Input Validation
Type check + content present — 400 if not
The first gate validates that the request body contains a type field (must be "idea" or "url") and a non-empty content field. Anything else gets an immediate 400 with a clear error code (INVALID_INPUT). This is defensive API design — fail fast, fail loudly, before touching any AI or storage.
5.2
Deduplication
inputHash — same idea never regenerated
Every input is normalized and hashed. Before running generation, the Data Table is queried by this hash. If a matching run exists, the existing state is returned immediately — with cached: true. This means submitting the same idea or URL twice never triggers another round of GPT-4o calls. Idempotent at the input level.
5.3
URL Ingestion
Jina.ai fetch → GPT-4o-mini summarise
If the input is a URL, Jina.ai's reader API fetches and extracts clean readable text from the page (handles SPAs, paywalls, JS-rendered content). This text is then passed to GPT-4o-mini with a content strategist prompt to produce a 150–220 word structured summary including core thesis, key points, and tone. This summary becomes the base idea for all downstream generation.
5.4
SEO Layer
SEO keyword extraction before article generation
Before generating articles, GPT-4o-mini generates structured SEO data: primary keyword, 5 secondary keywords, and a meta description. This runs as a separate step so the main article generation receives the SEO context and can incorporate keywords naturally. The output is strictly JSON — parse errors cause an explicit failure, not silent bad data.
5.5
Article Generation
GPT-4o generates three complete, distinct articles
The full GPT-4o model (not mini) generates three complete, distinct articles from the same base idea in a single call — each with a different angle, structure, and opening. The system prompt instructs it to behave as a senior SEO content writer. All three arrive as a JSON array which is parsed, and stored as draftA, draftB, draftC in the Data Table. The response includes all three drafts so the caller can present them for human selection.
Endpoint 2: /content-select-v4 — Draft selection + adaptation
POST /content-select-v4 → Validate executionKey + A/B/C → Get run by executionKey → Already adapted? → return cached → Build selected draft text → Adapt: LinkedIn + Tweet + Email (GPT-4o-mini) → Update content_runs row → Respond: content_ready
5.6
Selection + Caching
Select A, B or C — cached if already adapted
The caller submits their chosen draft (A, B, or C) with the executionKey. The workflow checks if adaptation already exists in the Data Table (linkedin, tweet, email all populated). If so, returns cached — no AI call made. If not, it pulls the chosen draft text and sends it to the adaptation step.
5.7
Multi-Platform Adaptation
One draft → LinkedIn post + tweet + email newsletter
GPT-4o-mini acts as a platform adaptation specialist. From the selected article, it produces: a LinkedIn post (professional tone, with hooks), a tweet thread (≤280 chars for the first tweet), and a newsletter email (with subject line and HTML-friendly body). All in one call, returned as JSON, parsed and stored per field in the Data Table.
Endpoint 3: /content-publish-v4 — Multi-platform publishing
POST /content-publish-v4 → Validate action (publish | schedule) → Already published? → cached → Schedule vs Publish? → LinkedIn already posted? → Post to LinkedIn → Twitter already posted? → Post to Twitter/X → Email already sent? → Send Gmail newsletter → status: published
5.8
Per-Channel Idempotency
Each channel has its own "already posted?" gate
This is the most carefully designed part of the system. Before posting to each platform, the workflow checks the per-channel publish status in the Data Table. If LinkedIn is already "posted", the LinkedIn step is skipped — but Twitter and email still run if they haven't fired yet. This means a partial publish failure (e.g. Twitter API is down) can be safely retried — only the channels that didn't complete will post again. No double-posting to any channel.
5.9
Schedule vs Publish
Schedule stores scheduledAt — publish fires immediately
The action field routes to either immediate publish (fires all three channels now) or schedule (stores a scheduledAt timestamp and status = "scheduled"). The schedule path responds immediately with confirmation — a separate scheduled workflow can pick these up at the right time. Both paths update the Data Table so the API layer always reflects current state.
Workflow 2 — Content Ops API (2 endpoints)
5.10
API Endpoint
POST /content-runs-list-v1 — paginated run history
Returns all content runs from the Data Table, sorted by updatedAt descending. The limit parameter is clamped to a maximum of 100 (Math.min logic) — no unbounded queries. Used by the frontend to show run history in the dashboard view.
5.11
API Endpoint
POST /content-run-get-v1 — single run lookup
Fetches a specific run by executionKey. Validates the key is present (400 if not), queries the Data Table, returns 404 with error code RUN_NOT_FOUND if it doesn't exist. Returns the full run state including all drafts, platform content, publish statuses, and timestamps. Clean, typed error responses — not generic 500s.
Edge cases I designed for
🔁
Idempotency everywhere
inputHash deduplication at entry, per-channel gates at publish, session checks at selection. Every endpoint is safe to call multiple times.
📡
Partial publish recovery
Per-channel publish status tracking means a failed publish (e.g. Twitter rate limit) can be retried without double-posting to channels that already succeeded.
🗃️
Durable state in Data Table
All state lives in the n8n Data Table — not in memory, not in the workflow execution. The API layer can query any run at any time regardless of when it was created.
Outcomes
Fetemi ends up with a true content system rather than a collection of writing tasks. One input can become multiple strong long-form options, a chosen angle can be adapted across channels, and the publish layer can be retried safely because the state is durable. The gain is not just speed. It is operational reliability.
